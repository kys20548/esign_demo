## 功能深化（呼應二面「漸進式翻新」提案，現場可以直接講）

- [x] **退回原因欄位**：主管按「退回」時**必填**原因，工程師在待簽清單可查看原因、並「重新填單」——原本的單保留成歷史紀錄，新單走同一個 Task 重新綁定
- [x] **簽核歷程時間軸**：詳情頁一律顯示「簽核歷程」卡片，從填單那一刻就記錄，之後核准／退回也會依序加進同一條時間軸；若同一派工有多次送單（例如退回後重新填），會把每次送單的填單、核准／退回事件全部串起來，依時間排列，目前這筆送單的事件會特別標示
- [x] **主管端「新增派工」表單**：`/admin` 頁有可收合的「＋ 指派新案場」表單，送出後立即出現在工程師的待簽清單（未填單狀態）
- [ ] **角色登入與權限**（暫緩，面試前不做）：目前任何人打開 `/admin` 就能核准單據——demo 為求簡潔省略，正式版必須加驗證，這點面試主動講出來反而顯示你知道 demo 和 production 的差距
- [x] **LINE 通知（模擬）**：核准/退回後 `/admin` 頂部會出現「📱 模擬 LINE 通知已送出給 XXX」橫幅，並註明「示意用，正式上線需申請公司 LINE 官方帳號串接 Messaging API」——誠實標示為模擬，避免誤導成真的已串接

## 工程紮實度（被問「這只是 demo 吧」時的底氣）

- [ ] **資料持久化**：目前純記憶體、重啟即清空；正式版要接 SQLite（單機夠用）或 PostgreSQL
- [ ] **基本單元測試**：至少涵蓋 store 的核心邏輯（新增、核准、異常統計）
- [ ] **CI（GitHub Actions）**：push 自動跑 `go vet` + `go build`，作品集加分、也是「Git 版控沒做好」這個痛點的直接示範
- [ ] **照片改存物件儲存**（S3 相容服務），而非塞進記憶體——公開 demo 長期跑會有記憶體壓力。查證（2026-07-15）：Render 自家的 S3 相容 Object Storage 還在 **Alpha 階段**（2026-03 開放候補名單，無定價），暫時不用；正式版可選 Render 內建（等 GA）或外部 Cloudflare R2 / AWS S3

## 加分巧思

- [ ] 首頁加「查看範例已核准單」快速連結，給不想操作的人也能一眼看到成品長相
- [x] PDF 匯出：詳情頁「🖨 匯出 / 列印 PDF」開一頁排版乾淨的列印稿（`/forms/{id}/print`），按「列印/儲存為 PDF」用瀏覽器原生功能存檔。**技術決策**：沒有手刻 Go PDF 二進位產生器，因為 PDF 標準字型不含中文字，要正確顯示繁體中文得手動內嵌 CJK 字型（Type0/CID font），複雜且容易做出空白或亂碼的檔案；改用瀏覽器列印機制，中文一定正確（瀏覽器自己有字型），也是很多正式系統的標準做法
