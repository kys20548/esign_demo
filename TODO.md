# 待辦與後續規劃

> 這份清單本身也是給台拓機電二面的道具——證明「先盤點、排優先級」不是口號。

## 面試前（本週，動作小、效益高）

- [ ] Render 保溫：設定 UptimeRobot 每 5 分鐘 ping 一次首頁，避免面試現場冷啟動 30–60 秒
- [ ] 手機個人熱點模式完整演練一次（怕面試現場沒有可用 Wi-Fi）
- [x] 產生一個指向線上網址的 QR code——面試時讓老闆**掃碼用自己手機操作**，比遞手機更自然，也更像「這是真的產品」而不是你在演示自己的作品。已完成：首頁「🔲 用手機掃碼開啟」→ `/qr` 頁面，QR 用 `qrencode` 產生（高容錯等級 H），已用 `zbarimg` 解碼驗證內容正確指向 https://esign-demo-40z9.onrender.com
- [ ] 表單送出失敗時的錯誤訊息優化（目前解析失敗只回傳陽春的純文字 400，體驗會扣分）

## 功能深化（呼應二面「漸進式翻新」提案，現場可以直接講）

- [x] **退回原因欄位**：主管按「退回」時**必填**原因，工程師在待簽清單可查看原因、並「重新填單」——原本的單保留成歷史紀錄，新單走同一個 Task 重新綁定
- [ ] **簽核歷程時間軸**：一張單從建立 → 簽核 →（若退回）重新送出的完整軌跡，稽核時好查
- [ ] **主管端「新增派工」表單**：現在的派工清單是寫死的種子資料，正式要用得能手動新增/指派客戶案場
- [ ] **角色登入與權限**：目前任何人打開 `/admin` 就能核准單據——demo 為求簡潔省略，正式版必須加驗證，這點面試主動講出來反而顯示你知道 demo 和 production 的差距
- [ ] **LINE 通知**：核准/退回後推播給工程師——機電業現場工班普遍用 LINE，比叫大家另外開一個 App 更貼近實際使用習慣，是個容易被老闆聽懂的加分點

## 工程紮實度（被問「這只是 demo 吧」時的底氣）

- [ ] **資料持久化**：目前純記憶體、重啟即清空；正式版要接 SQLite（單機夠用）或 PostgreSQL
- [ ] **基本單元測試**：至少涵蓋 store 的核心邏輯（新增、核准、異常統計）
- [ ] **CI（GitHub Actions）**：push 自動跑 `go vet` + `go build`，作品集加分、也是「Git 版控沒做好」這個痛點的直接示範
- [ ] **照片改存物件儲存**（S3 相容服務），而非塞進記憶體——公開 demo 長期跑會有記憶體壓力

## 加分巧思

- [ ] 首頁加「查看範例已核准單」快速連結，給不想操作的人也能一眼看到成品長相
- [ ] PDF 匯出：機電業許多稽核情境仍要紙本留存，核准後可下載存查用 PDF
